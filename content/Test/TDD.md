tdd는 무엇인가?
​
구현을 먼저 하고 테스트 하는 것이 아닌 테스트를 하고 구현을 하는 것이다.
​
테스트를 한다는 것은 기능이 올바르게 동작하는지 검증하는 테스트 코드를 작성한다는 것을 의미하며 이러한 테스트 코드를 먼저 작성한 뒤 통과시키기 위해 개발을 진행하는 것이 tdd이다.

___

테스트 코드는 어떻게 구현 하는가?
​
구현을 할 기능(함수 등)을 실행했을 때의 결과값이

내가 기대하는 결과값과 일치하는지 검증하는 코드를 만드는 것이다.

(PL별 적합한 방식으로)

내가 읽은 책에서는 아직 구현하지 않은 class와 메서드명을 호출하는 코드를 미리 적어 놓았다.
​
덧셈 기능을 테스트하는 코드를 작성한다고 했을 때

class Calculator와 method plus를 미리 작성한 것이다.

이렇게 함으로 다음과 같은 고민을 하게 될 것이다.
​
class의 이름은 뭘로 정할까?

메서드의 이름은 plus가 좋을까 sum이 좋을까

이 메서드의 파라미터는 타입이 뭐고 몇 개여야 할까?

return은 뭘로 하는 것이 좋을까?

​
테스트 코드를 미리 작성한다는 것이 결코 번거로운 일이 아닌 것 같다는 생각이 들었다.

단지 class나 함수 작성 시 우리가 보편적으로 하는 고민을 미리하는 것 뿐이다.
​
그리고 이렇게 만든 테스트 코드의 검증을 통과할 수 있게 기능 구현을 해나가면 되는 것이다.
​
___
​
tdd는 왜 좋은가?
​
바로 드는 생각은 솔루션이 커졌을 때 디버깅하는 속도와 사이드 이펙트를 고려하는 속도가 tdd를 사용하지 않았을 때에 비해 훨씬 빠를 것 같다.
​
특정 기능을 수정 할 때 사이드 이펙트를 고려하다 보면 이전에 이미 했던 생각도 다시 고려하게 되는 상황을 자주 겪었던 것 같은데 tdd를 사용하면 내가 수정하려는 기능의 설계 단계에서 부터 고려했던 다양한 케이스 별 테스트 코드를 작성해놨기 때문에 기존에 내가 생각해놨던 케이스는 테스트를 통과하기만 해도 되는 것이다. 또한 코드를 작성한 본인이 아니여도 이를 확인하기가 편하다.
​
유지보수면에서 tdd를 사용하지 않았을 때에 비해 압도적으로 탁월한 것이다.

---
​
테스트 코드를 점진적으로 구현해야한다.
​
가장 쉽거나 가장 예외적인 상황 부터 시작해서 그렇지 않은 상황으로 순차적이게 테스트 코드를 만들고 이를 통과하기 위해 점진적으로 기능을 구현해나가야한다.
​
이때 기능을 한번에 구현하는 것이 아닌 내가 작성한 테스트 코드를 통과할 정도로만 간단하게 구현하는 것이 포인트다.
