**통신 프로토콜과 계층 구조**

**통신 프로토콜의 정의**

- 통신 장치들간에 교환될 메세지의 형식 정의(Syntax)
- 메시지 교환 순서 정의(Timing)
- 메시지를 교환할 때 수행해야할 행위를 정의(Semantics)

하나의 프로토콜이 위 3가지를 정의 하며 프토토콜은 계층 구조 형태를 띈다.

**계층 구조**

- 프로토콜을 사용하는 User는 상위 계층의 프로토콜일 수도 있고 Human Interface 혹은 Server일 수도 있다.

 최종 사용자는 간 단계 마다의 프로토콜을 전부 사용하여 서비스를 받는 것이다.

그렇다면 왜 위 같이 계층적으로 포로토콜을 정의하는가?

**계층 구조의 장점**

- 새로운 프로토콜 정의 용이
- 특정 통신 기능 또는 서비스 수정 용이
- 전체 시스템 이해 용이

우리가 프로그래밍할 때 기능 별로 혹은 단계 별로 함수를 정의 하면 추후에 알아보기도 편하고 비슷한 기능이 필요할 때 또 가져다 쓰기 편하 듯이 프로토콜 또한 계층 구조로 정의 함으로써 비슷한 이점(재사용 가능, 유지보수 편리)을 갖는 것이다.

**계층 구조의 단점**

- 최적 시스템 구현의 어려움(프로토콜 단위 사용, 계층간 서비스 인터페이스 구현)

상위 계층의 프로토콜을 만들 때 하위 프로토콜에 내가 필요한 기능이 하나만 있는데 전부 다 가져다 써야하며 계층 마다 서비스 인터페이스를 정확히 분리해야 한다는 점에서 용량 측면에서 최적의 시스템 구현이 어렵다는 단점이 있다.

**계층 구조 모델**

실제로 우리가 인터넷에서 사용하는 계층 구조에 대해서 말씀 해주셨다.

**TCP/IP 계층 구조**

- 인터넷의 프로토콜 구조
- 5계층 구조

**OSI(Open System Interconnection) 구조**

- ISO(International Standardization Organization)에서 정의
- 7계층 구조
- 실제 사용되지 않고 참조 모델로 활용

TCP/IP 계층구조

| 계층 5 | application |
| --- | --- |
| 계층 4 | transport |
| 계층 3 | network |
| 계층 2 | link |
| 계층 1 | physical |

최상위는 Human Interface라고 볼 수 있다고 하셨다.

- 계층 1 (Physical)
    
    물리 계층, 정보를 전달하는 물리 매체를 정의하는 계층
    
- 계층 2 (Link)
    
    물리 매체를 이용해서 종단 장치 간의 패킷을 전송하는 절차를 정의하는 계층 
    
- 계층 3 (Network)
    
    링크들 간의 최적의 패킷 교환을 정의하는 계층
    
- 계층 4 (Transport)
    
    손실된 패킷의 복구 등 최종 사용자로 부터 송신될 패킷이 목적지 까지의 안전한 전달 절차를 정의하는 계층
    
- 계층 5 (application)
    
    최종 사용자가 필요로하는 정보를 교환하는 규칙을 정의하는 계층
    

OSI 계층구조

| 응용 | application | 계층 7 |
| --- | --- | --- |
| 프리젠테이션 | presentation | 계층 6 |
| 세션 | session | 계층 5 |
| 트랜스포트 | transport | 계층 4 |
| 네트워크 | network | 계층 3 |
| 링크 | link | 계층 2 |
| 물리 | physical | 계층 1 |

<aside>
💡 내가 알고 있는 계층 구조와 달라 찾아보니 TCP/IP를 5계층 구조로 정의 했을 때 위 처럼 정의를 하고 4계층 구조로 정의할 땐 Application - Transport - Internet - Link 로 정의하는 것 같은데 이 마지막 계층 1에 대해서도 말이 다 다른 것 같다.
누구는 Network Access, Network Interface, 누구는 Data Link, Link 등등 그냥 물리 영역(인터넷 전 단계) 까지의 영역을 계층 1로 보면 될 것 같다.

</aside>
