이번엔 git add라는 명령어에 대해 알아보자.

  

add 명령어는 working directory에서 작업한 파일을 staging area로 보내기 위한 명령어다.

위 말이 이해가 안 간다면 아래 포스팅을 보고 명령어를 알기전에 기본 개념을 대략적으로 알고 넘어가자.

[[3. Git 기본 개념]]

**status로 변경이력 확인**

  

위 셋팅을 마친 다음 cmd에 git status 명령어를 입력해보자.

![](images/Git/6.PNG)

이렇게 내 작업의 상태를 확인할 수 있다.

master branch에 있고

commits는 아직 없고

untracked files(git history가 처음 본 파일들) 목록을 보여주고

(여기서는 방금 셋팅에서 만들어준 txt 파일들이다.)

  

자 이제 여기서 staging area로 옮기고 싶은 파일들만 add 해보자.

---

**git add**

  

먼저 git add test.txt로 test.txt 파일만 옮겨보자. 그리고 git status를 다시 입력 하자.

![](images/Git/7.PNG)

그러면 아까는 없던 줄이 추가 되어있다.

저기 new file: test.txt가 존재하는 저 위치가 바로 staging area인 듯 하다.

  

이 staging area에서 제거하기 위해서 저기 나와있는 명령어를 그대로 입력해보자.

![](images/Git/8.PNG)

그리고 다시 git status 해보면 test.txt 파일이 다시 Untracked files로 넘어와 있는 것을 볼 수 있다.

  

---

**git add 복수 파일 지정**

  

파일을 단일 지정하는 것 외에도 git add를 여러 방법으로 사용할 수 있다.

리눅스 명령어에 익숙한 분이라면 늘상 하던일이라 크게 와닿진 않을 것 같다.

  

git add *.확장자명

![](images/Git/9.PNG)

우리는 txt 파일을 만들었으니 git add *.txt 명령어를 입력 후  
git status를 확인 해보면 위 처럼 모든 txt 파일들이 staging area로 넘어간 것을 확인할 수 있다.

  

![](images/Git/10.PNG)

unstaging도 똑같이 가능하다.

  

이번엔 txt 외의 다른 파일도 만들어보자 나는 test4.py 파일을 만들었다. 그런 다음

git add . 명령어를 입력하면

![](images/Git/11.PNG)

working directory에 있던 모든 변경 혹은 추가 내역들을 staging area로 전부 옮겨준다.

  

![](images/Git/12.PNG)

반대의 경우 그냥 . 으로는 안 되고 *.*로 지정하니 먹힌다.

  

commit은 또 따로 포스팅으로 다룰 것 같긴 한데 우선 commit을 한번 해보자.

![](images/Git/13.PNG)

commit 전에 당연히 git add로 staging area로 넘겨준 다음 하길 바란다.

그리고 까먹고 config의 email이랑 name 설정을 안 했더니 오류가 났다.

  

다음 git status를 입력 해보면

![](images/Git/14.PNG)

아무 것도 없다? No commits yet은 그럼 왜 보여줬던 거지?

  

git log --name-only 라는 명령어로 내가 원했던 내역을 확인할 수 있긴 하다.

![](images/Git/15.PNG)

git log에는 나오는데 원래 git status 명령어로는 commit한 파일 내역들을 볼순 없는건가 싶다.

  

아무튼 내가 보여주고 싶었던 것은 Untracked files이 없는 상태에서의 git status였다.

test.txt에 아무 내용이나 추가 해주고 git status를 다시 입력 해보자.

![](images/Git/16.PNG)

git status 출력 값이 아까랑 비슷한데 미묘하게 다르다.

  

아까는 모든 파일들이 **새로 생성**된 상태였기에 **Untracked files** 목록에 나열되어 있었고

지금은 commit을 한번한 뒤 **기존 파일을 수정**한 상태이기에 **modified:** 목록에 존재하는 것이다.

  

---

 **느낀점**

  

git add와 git status 명령어 이 두가지만 해도 하나의 포스팅 분량이 나오는게 참 신기할 따름이다.

그리고 생각보다 git status가 너무 친절하게 나와서 명령어로 git을 사용해도 큰 무리가 없을 것 같다는 생각도 든다.

물론 프로젝트 규모가 커지게 되면 commit 내역 같은 history들을 확인하기에는 UI툴 없이는 힘들 것 같긴 하다.

모쪼록 git 공부하는 것도 재밌게 느껴져서 다행이다.
