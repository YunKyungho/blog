**계층 구분**

(대략적인 개념)

  

git에는 크게 3가지 작업 환경으로 나뉘어져 있다.

  

1. **working directory**  
      
    우리가 실제로 파일을 수정하는 등 작업하고 있는 구역  
      
    
2. **staging area**  
      
    버전 히스토리에 저장하기 전 저장할 파일들을 옮겨 놓는 구역  
      
    
3. **git directory**  
      
    버전의 히스토리를 가지고 있는 Remote Repository(Github 등) 혹은 Local Repository

Working Directory에서 작업한 파일 중 저장할 파일을 add 명령어를 통해 Staging Area로 보낸다.

Staging Area에서 commit을 통해 Local Repository(Local Directory)으로 보낸다.

checkout으로 Local Repository에서 버전을 다시 불러올 수 있다.

(Merge는 일단 넘어가자.)

  

여기까지는 전부 내 로컬에서 일어나는 일들이다.

  

위 계층 구분 목차에서 git directory는  Remote Repository(Github 등) 혹은 Local Repository 2가지가 존재하는데

Local Repository에 존재하는 버전 히스토리가 날아가는 것을 대비하여 (혹은 협업을 위해)

push로 Remote Repository에 보내는 것이다. 반대로 가져오는 것은 pull로 한다.

  

> 수 많은 commit에서 버전 구분은 어떻게 하는가?

  

각각의 commit에는 고유한 해쉬코드가 존재하며 이를 통해 버전 정보를 참조한다. 이뿐만 아니라 commit에는 어떤 버전인지에 대한 message, 작성자, 작성날짜에 대한 정보도 함께 존재한다. 이런 정보들로 버전들을 구분할 수 있다.

---

**Working directory 상세**

  

working directory는 2가지로 구분지을 수 있다.

  

1. **untracked**  
    기존 히스토리에 존재하지 않아 git이 모르는 파일  
      
      
    
2. **tracked  
    **git이 이미 아는 파일을 의미하며 이는 또 2가지로 나뉜다.  
      
    1. unmodified  
       수정이 되지 않은 파일  
      
    2. modified  
       수정된 파일

당연하겠지만 새로 생겼거나 수정된 파일에 한해서만 add를 통해 staging area로 보낼 수 있다.**계층 구분**
